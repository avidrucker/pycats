# a11y / UX audit — menus, prompts, and text (#346)

**Date:** 2026-07-05 · **Agent:** ELDERBERRY · **Role:** RESEARCHER/REVIEWER · **Scope:** audit only — no code changed, follow-ups **proposed not filed**.

A pragmatic (not certified-WCAG) accessibility/UX pass over the game's menus, prompts,
and text, judged against the eight dimensions in #346. Contrast ratios use the WCAG
relative-luminance formula; the AA targets are ~4.5:1 (body) / ~3:1 (large ≥24px).

## Method

Read the menu/screen surfaces (`main_menu.py`, `options_menu.py`, `char_select.py`,
`pause_menu.py`, `win_screen.py`, `screen_manager.py`), the text/render path
(`text_utils.py`, `render_battle.py`), and the color/size constants (`config.py`).
Colors and sizes are cited by symbol name per the repo's anchor convention.

## Priority summary

notice / blocks comprehension; **P2** = noticeable, situational; **P3** = polish / edge.

## What's already handled well (credit where due)

- **Focus is not color-only.** `menu_widgets.FOCUS_MARKER = "►"` (U+25BA) is prepended to
  the focused row across `main_menu`, `pause_menu`, `options_menu` (rows + the Schemes row +
  the keybind-sets sub-menu list, `options_menu.py:535/546/631`), `text_entry`, and the
  char-select grid — each cites `#346` in-comment. So focus carries **three** redundant cues:
  the ► marker, a glow border (`BUTTON_BORDER_FOCUSED`), and yellow text. This closes most of
  the "state by color alone" risk the ticket worried about.
- **P1/P2 identity is redundant.** `P1_UI_COLOR` red (255,100,100) vs `P2_UI_COLOR` blue
  (100,100,255) is a colorblind-safe pair (not red/green), and every identity readout is also
  labeled with the literal text `"P1"`/`"P2"` (`render_battle.py:250`).
- **Menu text contrast is strong.** `MAIN_MENU_OPTION_COLOR` white on `MAIN_MENU_BG_COLOR`
  (20,20,30) ≈ **18:1**; focused YELLOW ≈ **17:1**. Char-select and win-screen backgrounds
  (30,30,40 / 40,40,50) are comparably dark. No menu-text contrast issue found.
- **The ► glyph is probe-guaranteed.** `text_utils._find_unicode_font` probes
  `["►","◄","↑","↓","✓","→","←"]` at startup and only selects a font that renders them, so the
  focus marker is safe **when any suitable font exists** (caveat: finding #3).

## Findings in detail

### 1 — P1 — Player-facing dev jargon in the HUD *(Wording)*
`render_battle.draw_hud` (`render_battle.py:856–868`) renders **7 lines always** — no dev
gate (contrast the hitbox overlay `render_battle.py:828`, which *is* toggle-gated). Two lines
are debug output, not player UX:
- `fsm = f"FSM: {p.state.capitalize()}"` — "FSM" is implementation jargon (finite-state machine).
- `shield_attempting = f"Shield Attempting: {'Yes' if ... }"` — an internal boolean.

`Shield HP:`, `Lives:`, `Damage:`, and jumps-left are legitimate player info, but the two above
read as a debugger. `game.py:201–202` already carries a `#### TODO` for a "dev info bool flag."
**Recommendation:** hide `FSM:`/`Shield Attempting:` (and any future internals) behind a
dev-info toggle defaulting **off**; keep the real stats always-on. Follow-up: `enhancement`,
depends on / subsumes the game.py dev-flag TODO.

### 2 — P1 — P2 cat barely separates from the battle background *(Contrast)*
`P2_COLOR = (90,90,90)` (gray body) on `BG_COLOR = (60,60,70)` ≈ **1.6:1**. The gray P2 cat is
low-contrast against the gray-blue stage — a visibility problem in motion, worse for low-vision
players. P1 orange (255,160,64) on the same bg is fine (~4:1+). This is sprite identity, at the
edge of the "text/prompt" scope, but it's the single biggest *perceptual* gap.
**Recommendation:** either darken/brighten `P2_COLOR` away from the bg luminance, or add a thin
contrasting outline to both fighters (helps P1 over bright stage regions too). Any color change
needs the RULES "changing a value needs a basis" gate — cite this audit's contrast measurement.
Follow-up: `enhancement`.

### 3 — P2 — Arrow/marker glyphs fall back to "?" without a Unicode font *(Legibility)*
`_find_unicode_font` degrades to `"No suitable Unicode font found, using ASCII fallbacks"`
(`text_utils.py:127`), but `render_text_mixed` renders any non-whitelisted glyph as `"?"`. On a
host lacking a font that covers `►/←/→/↑/↓`, the focus marker and the control-legend arrows
become `"?"` — focus feedback silently regresses to color-only, and controls read `Move(??)`.
Low probability (most desktops ship such a font) but total failure when it hits.
**Recommendation:** give `render_text_mixed` an explicit ASCII fallback map (`► → ">"`,
`← → "<"`, `↑ → "^"`, `↓ → "v"`) instead of `"?"`, so degraded rendering stays legible.
Follow-up: `enhancement`.

### 4 — P2 — HUD is a flat label stack with no hierarchy *(Wording / consistency)*
The seven HUD rows are equal-weight white text at one size (`HUD_FONT_SIZE`), top-left/right
stacked (`render_battle.py:864–868`). Damage % and stock count — the two numbers players read
constantly — get no more emphasis than the debug lines. No grouping, no size contrast.
**Recommendation:** a small layout pass — promote Damage/Lives (larger / corner-anchored, per
the existing `game.py:201` TODO), demote the rest. Follow-up: `enhancement` (pairs with #1).

### 5 — P2 — HUD text contrast is bg-dependent *(Contrast)*
HUD white text ≈ **11:1** over the flat `BG_COLOR`, but it's drawn over the **live** scene —
white P1 highlights, the light "ghost" cat skin (255,255,255), or a bright stage region can drop
local contrast below AA. No backplate or shadow today.
**Recommendation:** a 1px dark text shadow or a semi-opaque backplate behind HUD/percent text
makes it bg-independent. Follow-up: `enhancement` (small).

### 6 — P3 — Focused-row color is YELLOW-vs-WHITE *(Colorblind)* — **acceptable**
For blue-yellow CVD (tritanopia), yellow can read close to white, so the *color* delta of the
focused row is weak. **But** the ► marker + glow border already convey focus non-chromatically
(see "handled well"), so no action needed. Logged so a future refactor doesn't drop the marker
and reintroduce the risk.

### 7 — P3 — Toggle-label terminology is inconsistent *(Consistency)*
Options toggles mix noun styles and abbreviations — e.g. `Status Bars`, `Hitbox Overlay`,
`Input History`, `Controls` (`options_menu` row labels). Not wrong, but inconsistent
capitalization/verbness across a single screen.
**Recommendation:** a terminology sweep for parallel phrasing (all noun-phrases, consistent
casing). Follow-up: `enhancement` (docs/labels, low).

### 8 — P3 — Hold-to-act affordances *(Hidden affordance)* — verify
Hold-ESC-to-quit (#113) and hold-B-to-menu (#20) have progress timers (`screen_manager`
`esc_quit_timer` / `back_timer`, and a `MAIN_MENU_SELECTED_COLOR` progress draw at
`screen_manager.py:315`). The *progress* is shown once holding starts; open question is whether
the initial **hint** ("hold ESC to quit") is discoverable before the player knows to hold.
**Recommendation:** confirm a resting hint exists on the relevant screens; if not, that's a
small `enhancement`. Follow-up: research/verify (cheap).

## Termination / known-unknowns

- **Time-boxed** to ~60m; this is a pragmatic pass, not certified WCAG conformance.
- **Not measured live:** contrast ratios are computed from `config.py` constants, not sampled
  from rendered frames over real stage/sprite pixels (findings #2/#5 are bg-dependent — the
  numbers are the flat-bg baseline). A pixel-sampling pass would sharpen #5.
- **Font scaling (#345):** the FONT_SCALES machinery (small/standard/large, `config.py:425`)
  exists; this audit judged the *need* (met) not the implementation — deferred to #345 by scope.
- **char-select 2D focus (#416):** the grid uses ► + a white cursor rect (two cues) — this audit
  finds that adequate for the color-only concern; #416's open question is layout *consistency*
  with the 1D menu-button widget, which is a separate design thread, not an a11y defect.
